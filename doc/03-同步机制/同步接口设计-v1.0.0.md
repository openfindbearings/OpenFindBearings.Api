### 1. 接口定位

同步接口供 **FindBearings.Sync** 项目的 L 阶段（Load）调用，将清洗后的数据写入业务数据库。

认证方式：OAuth2 Client Credentials 流程，要求 `scope=api:sync` claim。

### 2. 端点列表

所有端点部署在 `/api/sync` 路由组下，`RequireAuthorization("SyncClient")`。

| 方法 | 路由 | 响应模式 | 请求体 |
|------|------|------|------|
| POST | `/api/sync/brands/batch` | 同步 200 | `{ Brands: SyncBrandDto[] }` |
| POST | `/api/sync/bearingtypes/batch` | 同步 200 | `{ Types: SyncBearingTypeDto[] }` |
| POST | `/api/sync/bearings/batch` | 异步 202 | `{ Bearings: SyncBearingDto[] }` |
| POST | `/api/sync/merchants/batch` | 异步 202 | `{ Merchants: SyncMerchantDto[] }` |
| POST | `/api/sync/merchantbearings/batch` | 异步 202 | `{ MerchantBearings: SyncMerchantBearingDto[] }` |
| POST | `/api/sync/interchanges/batch` | 异步 202 | `{ Interchanges: SyncInterchangeDto[] }` |
| GET | `/api/sync/tasks/{taskId}` | 查询状态 | - |

### 4. 处理逻辑

#### 4.1 Upsert 模式

所有同步端点为增量 Upsert：

1. 遍历请求数组中的每条记录
2. 按 Code / PartNumber 检查是否存在
3. 存在 → 更新；不存在 → 创建
4. 返回 `BatchResult`（SuccessCount + FailCount + SuccessIds）

#### 4.2 覆盖保护（Bearing 专用）

Bearing 同步 handler 有覆盖保护规则：

```
if (existingBearing.DataSource?.SourceType != DataSourceType.Crawler)
    → 跳过（保护非爬虫数据：纠错、人工导入、Excel 导入）
```

这要求 Sync 侧在调用时必须传递 `DataSource = "Crawler"`（或其他来源类型），否则已有记录不会被更新。

#### 4.3 关联解析

Bearing 同步时依赖品牌和类型已存在：
- `BrandCode` → 查询 `Brand.Code` → 不存在则失败
- `BearingTypeCode` → 查询 `BearingType.Code` → 不存在则失败

因此品牌和类型必须在轴承之前同步。

### 5. 已知问题

#### 5.1 异步任务追踪存根（已发现）

`GET /api/sync/tasks/{taskId}` 处理程序是存根，返回固定假数据。轴承、商家、关联、替代品的 202 Accepted 响应后，任务状态无法被正确轮询。

**影响：** LoadService 的 `WaitForTaskCompletionAsync` 永远超时。

**待处理：** 考虑将 4 个异步端点改为同步（与品牌/类型一致），消除假异步机制。

#### 5.2 DataSource 传递缺失（已发现）

`SyncBearingDto` 已定义 `DataSource` 和 `SourceSite` 字段，但 LoadService 侧尚未映射传递。

**影响：** 覆盖保护规则无法识别数据来源，所有同步数据均以默认值写入。
