### 1. 接口定位

同步接口供 **FindBearings.Sync** 项目的 L 阶段（Load）调用，将清洗后的数据写入业务数据库。

认证方式：OAuth2 Client Credentials 流程，要求 `scope=api:sync` claim。

限流豁免：`/api/sync/*` 路径在 `RateLimitingMiddleware` 中白名单放行，不受用户级限流影响。

### 2. 端点列表

所有端点部署在 `/api/sync` 路由组下，`RequireAuthorization("SyncClient")`。
全部为**同步 200 响应**，返回 `ApiResponse<BatchResult>`。

| 方法 | 路由 | 请求体 |
|------|------|------|
| POST | `/api/sync/brands/batch` | `{ Brands: SyncBrandDto[], Mode: SyncMode }` |
| POST | `/api/sync/bearingtypes/batch` | `{ BearingTypes: SyncBearingTypeDto[], Mode: SyncMode }` |
| POST | `/api/sync/bearings/batch` | `{ Bearings: SyncBearingDto[], Mode: SyncMode }` |
| POST | `/api/sync/merchants/batch` | `{ Merchants: SyncMerchantDto[], Mode: SyncMode }` |
| POST | `/api/sync/merchantbearings/batch` | `{ MerchantBearings: SyncMerchantBearingDto[] }` |
| POST | `/api/sync/interchanges/batch` | `{ Interchanges: SyncInterchangeDto[] }` |

### 3. 响应格式

```json
{
  "success": true,
  "code": 200,
  "message": "batch sync complete",
  "data": {
    "successCount": 45,
    "failCount": 3,
    "totalCount": 50,
    "items": [
      { "identifier": "6205", "action": "created", "id": "guid...", "error": null },
      { "identifier": "6309", "action": "skipped", "id": null, "error": "非爬虫数据，跳过覆盖保护" },
      { "identifier": "invalid", "action": "failed", "id": null, "error": "品牌不存在: XXX" }
    ]
  }
}
```

| 字段 | 说明 |
|------|------|
| `data.items[].identifier` | 实体标识：品牌/类型用 Code，轴承用 PartNumber，商家用 Name，关联/替换用复合键 |
| `data.items[].action` | `created` / `updated` / `skipped` / `failed` |
| `data.items[].id` | 成功创建/更新后的业务实体 ID |
| `data.items[].error` | 失败或跳过的原因 |

- `skipped` 状态的项不计入 `successCount` 也不计入 `failCount`
- `totalCount` = `successCount + failCount`（不包含 skipped）

### 5. 处理逻辑

#### 5.1 Upsert 模式

所有同步端点默认使用 `SyncMode.Upsert`：

1. 遍历请求数组中的每条记录
2. 按业务键（Code / PartNumber / Name）检查是否存在
3. 存在 → 更新字段；不存在 → 创建
4. 返回 `BatchResult`（含逐条 `Items` 结果）

#### 5.2 覆盖保护（Bearing 专用）

```csharp
if (existingBearing?.DataSource?.SourceType != DataSourceType.Crawler)
    → AddSkipped("非爬虫数据，跳过覆盖保护")
```

只允许 Crawler 来源的数据覆盖已有记录。Manual、FileImport、Api、SeedData、UserSubmitted 均受保护。SourceType 为 null 的旧数据（如早期种子数据）同样受保护。

#### 5.3 DataSource 类型映射

| Sync 传入 `DataSource` | 实际设置 `DataSourceType` |
|-------------------------|--------------------------|
| `"crawler"` | `Crawler`（站点名存入 ImportedBy） |
| `"manual"` | `Manual` |
| `"api"` | `Api` |
| `"fileimport"` / `"file"` | `FileImport` |
| `"seeddata"` / `"seed"` | `SeedData` |
| 其他 / null | `Manual`（默认） |

#### 5.4 关联依赖

Bearing 同步依赖品牌和类型已在 API 中存在：
- `BrandCode` → 查询 `Brand.Code` → 不存在则失败
- `TypeName` → 查询 `BearingType.Code` → 不存在则失败

Sync 侧在 L 阶段发送前会将类型中文名（如 "圆柱滚子轴承"）解析为 Code（如 "CRB"），品牌中文名（如 "天马"）解析为 Code（如 "TMB"）。

加载顺序：**品牌 → 类型 → 轴承 → 商家 → 关联 → 替代品**。

#### 5.5 验证规则

`BatchCreateBearingsCommandHandler` 中每条轴承创建前会通过 FluentValidation 验证：

| 规则 | 约束 |
|------|------|
| PartNumber | 必填，最长 100 字符，允许大小写字母、数字、`-/.( )+` |
| 尺寸 | InnerDiameter > 0, OuterDiameter > InnerDiameter, Width > 0 |
| 关联 | BearingTypeId + BrandId 必填 |
| 重量 | Weight > 0（若有值） |
| 动/静载荷 | DynamicLoad <= StaticLoad × 1.5（若两者都有值） |
| 精度 | 格式 P0-P6 或单数字 |
